# Classe abstraite en POO

## 1. Classe abstraite : définition & syntaxe

> **Définition** :  
Une **classe abstraite** est une classe **incomplète** qu'on ne peut **pas instancier**.  
Elle sert de **modèle (*template*) partiel** pour d'autres classes qui vont la **compléter**.  
Sa déclaration est accompagnée du mot-clé `abstract`.

Une classe qui n'est pas abstraite, donc instanciable avec `new`, est dite **concrète**.

```java
public abstract class MaClasseAbstraite {
    // Implémentation de la classe abstraite
}
```

### Points essentiels : classe abstraite

- Contient du **code commun** (attributs, méthodes) définissant un modèle incomplet pour ses éventuels descendants.
- Peut contenir des **méthodes abstraites** (sans corps) **ou non**. Cependant, si une classe contient au moins une méthode abstraite, elle doit être déclarée `abstract`.
- Ne **peut pas être instanciée** : `new MaClasseAbstraite()` → ❌ Erreur.
- Les sous-classes **doivent implémenter** les méthodes abstraites (ou demeurer elles-mêmes `abstract`).
- Peuvent avoir des **constructeurs**, même si elles ne sont pas instanciables. Ces constructeurs sont appelées par les constructeurs des sous-classes (chaînage).

### Méthode abstraite

> **Définition** :  
Une **méthode abstraite** est une méthode **sans implémentation** (sans corps).  
Au lieu des `{ }` contenant son implémentation, sa déclaration contient la mention `abstract` et se termine par un `;`.

```java
// Exemple de méthode abstraite
public abstract void maMethode(int param1, String param2);
```

#### Caractéristiques : méthodes abstraites

- Peut être déclarée dans une classe qui est déjà abstraite. Si une méthode est abstraite, **elle rend sa classe abstraite** (si elle ne l'était pas).
- Oblige les classes dérivées à fournir **leur propre version** de la méthode.
- Ne **peut pas être `private`**, sinon elle est impossible à redéfinir dans les sous-classes (inaccessible).

### Exemple de classe abstraite

```java
public abstract class Fruit {
    protected double masse; // ok de mettre protected, car classe abstraite

    public abstract void gouter(); // Pas de corps = abstraite

    public void afficher() {       // Méthode normale
        System.out.println("Je suis un fruit.");
    }
}
```

## 2. Cas d'utilisation : classe abstraite

On utilise une classe abstraite quand :

- **On veut partager du code commun**  
  *Ex*. : tous les fruits ont une masse, un goût, une méthode `afficher()`.
- **On veut forcer les sous-classes à implémenter certaines méthodes**  
  *Ex*. : chaque fruit doit définir `gouter()`.
- **On veut un comportement générique, mais impossible d'en faire un objet complet**  
  *Ex*. : un `Fruit` ne représente rien de **concret** tout seul.

## 3. Interface : définition & syntaxe

> **Définition** :  
Une **interface** est un **contrat** qui définit **ce qu'une classe doit faire**, mais **pas comment** elle le fait.

Les interfaces sont déclarées similairement aux classes, mais en remplaçant `class` par le mot réservé `interface`.

```java
public interface Poids {
    double G = 9.80665; // Attribut, implicitement `public static final`
    double getPoids();  // Méthode sans implémentation, implicitement `public` et `abstract`
    default void aUnPoids() { System.out.println("J'ai un poids!"); }       // Méthode `default`, implicitement `public`
    static void acceleration() { System.out.printf("g = %f m/s^2", G); }    // Méthode `static`, implicitement `public`
}
```

### Points essentiels : interface

- Elle peut contenir :
  - des **méthodes sans corps** elles sont implicitement `abstract`.
  - des **méthodes `default`** (avec un corps, depuis Java 8).
  - des **méthodes `static`** (depuis Java 8).
- **Toutes** les méthodes des interfaces sont implicitement `public`, alors il est optionnel de le spécifier.\*
- Elle **ne peut pas contenir d'attributs d'instance**, seulement des **constantes** (attributs `public static final`).

\* *Depuis Java 9, il est possible d'avoir des méthodes d'interface de visibilité* `private`*, mais elles sont exclues de ce cours*.

## 4. Cas d'utilisation : interface

On utilise une **interface** quand on veut définir une **capacité** ou un **comportement** que plusieurs classes différentes peuvent avoir, **peu importe leur parent commun**.

### Cas d'utilisation courants

#### A) Décrire une capacité

**Exemples** :

- `Comparable` : capacité d'être **ordonné** (donc trié)
- `Runnable` : capacité d'être **exécuté** (au sens informatique!)
- `Serializable` : capacité d'être **sauvegardé**
- La capacité de **voler** : un avion, un oiseau ou un drone peuvent l'implémenter sans être liés par l'héritage.

    ```java
    public interface Volant {
        void voler();
    }
    ```

#### B) Permettre l'implémentation multiple

Java ne permet **d'hériter d'une seule classe**, mais permet **l'implémentation de plusieurs interfaces**.

```java
public class Avion implements Comparable<Avion>, Volant, Serializable {
    public void voler() { System.out.println("Je vole!"); } // doit être implémentée sinon Avion doit être abstract
}
```

#### C) Définir une API cohérente

Utile dans des projets où certaines classes doivent offrir la même méthode, mais **n'ont aucun lien hiérarchique**.

#### D) Remplacer certaines classes abstraites légères

Si une classe abstraite ne contient **pas d'état** (attributs), une **interface** suffit souvent.

## 5. Comparatif : classe abstraite vs. interface (et quand choisir l'une ou l'autre)

|Aspect|Classe abstraite|Interface|
|---|---|---|
|Instanciation|**Non**|**Non**|
|Constructeur(s)|**Oui**|**Non**|
|Héritage|Héritage **simple** (une seule classe mère)|**Implémentations multiples** possibles (une classe peut implémenter plusieurs interfaces)|
|Attributs d'instance (état)|**Oui**|**Non** (seulement constantes `public static final`)|
|Code commun|**Oui**, méthodes concrètes + champs|**Oui**, via méthodes default/static (mais sans état d'instance)|
|Objectif|**Modèle de base** avec code + état commun|**Contrat** (capacité/comportement) à respecter|

### Comment choisir?

- Besoin d'état commun, de constructeurs, d'un template : **classe abstraite**.
- Besoin d'exprimer des capacités (ex. `Comparable`, `Serializable`) : **interface**.
- Tu peux **combiner** : une hiérarchie avec une classe abstraite de base + plusieurs interfaces pour les capacités transversales.
